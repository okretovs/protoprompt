# Triage

Move Linear issues through a small state machine of triage roles.

Linear is the issue tracker. GitHub PRs are implementation artifacts linked to Linear issues; they are not the triage source of truth unless the user explicitly asks to review a PR.

Every comment posted to Linear during triage **must** start with this disclaimer:

```markdown
> *This was generated by AI during triage.*
```

## Reference docs

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — how to write durable implementation briefs
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — how the `.out-of-scope/` knowledge base works
- `docs/agents/issue-tracker.md` — Linear MCP workflow
- `docs/agents/triage-labels.md` — mapping from canonical roles to Linear statuses/labels

## Roles

Two **category** roles:

- `bug` — something is broken
- `enhancement` — new feature or improvement

Five **state** roles:

- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter or user clarification
- `ready-for-droid` — fully specified, ready for Factory Droid implementation
- `ready-for-human` — needs human implementation or judgment
- `wontfix` — will not be actioned

Every triaged Linear issue should carry exactly one category role and one state role, expressed using the Linear status/label mapping in `docs/agents/triage-labels.md`. If state roles conflict, flag it and ask the maintainer before changing anything.

The mapping should have been provided to you. Run `/setup-project-skills` if it is missing.

State transitions: an untriaged issue normally goes to `needs-triage` first; from there it moves to `needs-info`, `ready-for-droid`, `ready-for-human`, or `wontfix`. `needs-info` returns to `needs-triage` once the reporter replies. The maintainer can override at any time; flag unusual transitions and ask before proceeding.

## Invocation

The maintainer invokes `/triage` and describes what they want in natural language. Interpret the request and act. Examples:

- "Show me anything that needs my attention"
- "Let's look at ABC-123"
- "Move ABC-123 to ready-for-droid"
- "What's ready for Droid to pick up?"

## Show what needs attention

Query Linear and present three buckets, oldest first:

1. **Untriaged** — no mapped state role yet.
2. **`needs-triage`** — evaluation in progress.
3. **`needs-info` with reporter activity since the last triage notes** — needs re-evaluation.

Show counts and a one-line summary per issue. Let the maintainer pick.

## Triage a specific Linear issue

1. **Gather context.** Read the full Linear issue through MCP: body, comments, labels/status, author, dates, project/cycle, priority, relations, and links. Parse any prior triage notes so you do not re-ask resolved questions. Explore the codebase using the project's domain glossary, respecting ADRs in the area. Run two checks against the codebase: (a) **redundancy** — search for an existing implementation of the requested behavior by domain concept, not just the request's wording, and report where you looked. If found, it is an already-implemented `wontfix` candidate. (b) **prior rejection** — read `.out-of-scope/*.md` and surface any entry that resembles this request.

2. **Recommend.** Tell the maintainer your category and state recommendation with reasoning, plus a brief codebase summary relevant to the request, including whether it is already implemented. Wait for direction.

3. **Verify the claim.** Before clarifying requirements, check that the claim holds up. For a bug, reproduce it from the reporter's steps. If a linked GitHub PR exists and the user asked to consider it, inspect the diff and run the relevant tests or commands. Report what happened: confirmed, failed, or insufficient detail. A confirmed verification makes a much stronger implementation brief.

4. **Clarify if needed.** If the request needs fleshing out, run `/grilling` and `/domain-modeling` together: ask one question at a time, sharpen domain terms, and update `CONTEXT.md` or ADRs as decisions land.

5. **Apply the outcome using Linear MCP:**
   - `ready-for-droid` — post an implementation brief comment ([AGENT-BRIEF.md](AGENT-BRIEF.md)) and apply the mapped Linear status/label.
   - `ready-for-human` — same structure as an implementation brief, but note why it cannot be delegated: judgment calls, external access, design decisions, or manual testing.
   - `needs-info` — post triage notes using the template below.
   - `wontfix` — close or mark with the mapped Linear status/label, with the comment depending on why:
     - **Already implemented** — the change already exists in the codebase. Point to where it lives; do not write to `.out-of-scope/`.
     - **Rejected bug** — give a polite explanation, then close or mark accordingly.
     - **Rejected enhancement** — write to `.out-of-scope/`, link to it from a comment, then close or mark accordingly ([OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)).
   - `needs-triage` — apply the mapped Linear status/label. Optional comment if there is partial progress.

## Quick state override

If the maintainer says "move ABC-123 to ready-for-droid", trust them and apply the mapped Linear role directly. Confirm what you are about to do, then act. Skip grilling. If moving to `ready-for-droid` without a clarification session, ask whether they want an implementation brief.

## Needs-info template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

Capture everything resolved during clarification under "established so far" so the work is not lost. Questions must be specific and actionable, not "please provide more info".

## Resuming a previous session

If prior triage notes exist on the Linear issue, read them, check whether the reporter has answered any outstanding questions, and present an updated picture before continuing. Do not re-ask resolved questions.
